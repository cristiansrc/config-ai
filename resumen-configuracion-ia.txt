Política de Estado de Specs
- Toda spec debe declarar estado cerca del inicio del archivo.
- Estados editables: planning, draft, validated-not-executed.
- Estados no editables: executed, implemented, closed, superseded.
- Si una spec ya fue ejecutada/implementada/cerrada, no se modifica en sitio.
- Cualquier cambio posterior debe crearse como nueva spec incremental en docs/specs/increments/ y debe referenciar la spec original.
- Planner es responsable de aplicar cambios de planificación/specs y de actualizar OpenAPI cuando corresponda.
- Planner debe asegurar un `.gitignore` en la raiz del repositorio activo cuando cree o actualice specs. Debe cubrir archivos no versionables de todo el proyecto: artefactos temporales de specs como `docs/specs/.working/`, secretos/env, logs, dependencias, outputs de build, coverage/tests, runtime local, IDE/OS y artefactos generados por stacks Java/Spring, Node/React/Angular, Python, n8n o Docker segun aplique. No debe ignorar specs incrementales, OpenAPI, migraciones ni documentacion versionable.
- Spec Validator debe bloquear cambios directos sobre specs ya ejecutadas.

Política del Executor
- Executor usa qwen3.5-9b-claude-4.6-opus-reasoning-distilled-v2 para priorizar velocidad en implementación de specs ya validadas.
- Durante codificación no debe pedir aprobación después de cada archivo creado o modificado.
- Solo debe detenerse para preguntar si hay duda bloqueante, contradicción, decisión faltante, acción insegura o cambio de alcance requerido.
- Si no hay bloqueo, debe completar implementación, tests/verificación y luego reportar archivos cambiados, resultados y riesgos residuales.

Optimización del Proceso SDD
- Planner debe producir un set consistente por incremento: spec/delta, OpenAPI si aplica, contrato de migraciones, contrato de integración/configuración y Decomposition Contract.
- Spec Validator debe validar no solo la spec, sino también coherencia contra OpenAPI, migraciones, config/realm y task decomposition existente.
- Task Decomposer debe leer el Decomposition Contract y bloquear si faltan fuentes canónicas o si detecta nombres/rutas/headers/status viejos.
- Executor debe usar prioridad de fuentes: petición actual, artefactos implementados del repo, specs/tareas activas, históricos solo como trazabilidad.
- Si task decomposition contradice OpenAPI, migraciones, config o spec validada, Executor debe detenerse con `Blocked: artifact mismatch` en vez de implementar.
- No se debe pasar a implementación si `task-decomposition.md` contiene términos obsoletos de iteraciones anteriores.
- Planner y Spec Validator deben compartir contexto persistente mediante `docs/specs/.working/<increment>-sdd-context.md`.
- Task Decomposer debe crear un task board persistente en `docs/specs/tasks/<increment>-task-board.md`.
- Executor debe marcar tareas como `todo`, `in_progress`, `done` o `blocked`, con razón y artefactos en conflicto cuando aplique.
- Todo agente que cree archivos debe crear y verificar primero el directorio padre; solo después debe escribir el archivo.
- Task Decomposer y Executor solo pueden avanzar cuando shared context/spec están `validated-not-executed` y el último Spec Validator dice `ready`.
- Planner no puede invocar, recomendar ni establecer como siguiente acción Task Decomposer/Executor sin aprobación explícita de Spec Validator registrada en shared context.
- La aprobación de Spec Validator debe estar en el shared context como heading exacto `## Spec Validator Approval`, con `verdict: ready`, `reviewed_at`, `validator_agent: spec-validator`, `artifact_set_reviewed`, `summary` e `invalidated_by_changes_since: none`.
- Frases narrativas como `Ready for Task Decomposer`, `Artifacts aligned` o `all findings resolved` no cuentan como aprobación.
- Spec Validator puede editar solo reportes de validación y metadatos de estado/lifecycle: `spec-validation.txt`, `spect-vaidation.txt` si ya se está usando, `Current status`, `Spec Validator Approval`, status principal de spec y estados del task board. No puede editar contratos técnicos, OpenAPI, migraciones, tareas ni código.
- Al terminar cada validación, Spec Validator debe actualizar el reporte de validación, mover findings viejos a `Superseded findings` cuando ya no apliquen y dejar estados coherentes con `ready` o `not ready`.
- Si cualquier review summary o Spec Validator dice `not ready`, la siguiente acción obligatoria es Planner corrections + nueva validación.
- Queda prohibido usar `Known Technical Debt`, `Override Approved by User` o `fix post-increment` para saltarse drift entre spec, OpenAPI o migraciones, salvo aprobación explícita del usuario registrada en shared context.
- Solo debe existir un shared context activo por incremento; contexts anteriores deben marcarse `superseded` o históricos.
- Los checklists de consistencia deben verificar archivos reales e incluir evidencia de ruta, campo/endpoint/columna revisada y resultado observado.
- Cada ruta en `Canonical artifacts` debe existir realmente; el agente debe leer archivos o listar directorios antes de marcar consistencia.
- El task board solo puede usar estados `todo`, `in_progress`, `done` o `blocked`; si hay `Blocker:` o estado superior `blocked`, no pueden quedar tareas de implementación ejecutables en `todo`.
- La ruta de migraciones Flyway debe coincidir con la configuración real: `src/main/resources/db/migration` para classpath o `db/migration` solo si Flyway usa `filesystem:db/migration`.
- OpenAPI y spec deben usar el mismo schema de errores REST. Para Spring Boot, la estructura estándar es `timestamp`, `status`, `error`, `code`, `message`, `path`, `trace_id` y `details`.
- Después de compaction o sesión resumida, Planner, Spec Validator, Task Decomposer y Executor deben rehidratar contexto desde disco: shared context, spec, OpenAPI, migraciones/config, task board y último reporte de validación. El resumen compacto del chat solo es una pista.
- Si un reporte viejo o resumen contradice archivos actuales, se marca como `superseded` y se usan los archivos actuales.
- Nuevo agente `spec-remediator`: usa qwen3.6-27b para corregir hallazgos de Spec Validator uno por uno. Planner sigue siendo dueño de specs, arquitectura, alcance y contratos. Spec Remediator solo corrige hallazgos mecánicos o drift con fuente autoritativa inequívoca; decisiones de diseño se enrutan al Planner/User.
- Spec Remediator primero solicita validación completa, clasifica hallazgos (`mechanical`, `contract-drift`, `design-decision`, `migration-risk`, `validator/process-bug`, `user-decision`), toma el primer hallazgo seguro, aplica una corrección mínima, pide validación solo de ese hallazgo y continúa. Máximo 4 intentos por hallazgo; si no puede resolverlo o reaparece un hallazgo ya resuelto, detiene el flujo y escribe reporte en `/home/cristiansrc/Documentos/config-ai/bugs/`.
- `spec-remediator` no puede llamar a Task Decomposer ni Executor; solo trabaja contra Planner/Spec Validator y artefactos SDD.
- Estos artefactos se crean por proyecto/incremento durante el flujo SDD; no debe existir un task board activo global por defecto.
- Nuevo agente `requirements-analyst`: usa lmstudio/qwen/qwen3.6-35b-a3b antes de Planner para levantar requerimientos cuando la solicitud aun no esta lista para SDD formal.
- Entregable de requirements-analyst: `docs/specs/requirements/<increment-or-feature>-requirements-brief.md` o, si el proyecto usa contexto temporal, `docs/specs/.working/<increment-or-feature>-requirements-brief.md`.
- El requirements brief debe incluir estado `requirements-discovery`, objetivo, contexto, actores, roles/permisos, alcance, no objetivos, flujos, entidades funcionales, integraciones, seguridad, restricciones operativas, edge cases, criterios de aceptacion, preguntas abiertas y handoff para Planner.
- `requirements-analyst` no debe escribir OpenAPI, migraciones, task boards, specs incrementales formales ni codigo. Tampoco debe tomar decisiones tecnicas profundas; debe preparar informacion y marcar decisiones pendientes para Planner/User.
